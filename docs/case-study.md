# Case study: Brightpath, or how 56 alerts turn out to be 6 decisions

This is the walkthrough I give when demoing The Watcher. Brightpath is the
built-in demo company — 20 services, fake fintech, generated by
`watcher ingest --demo`. Nothing in it is exotic; every problem planted in
it is something I've personally watched happen at real companies.

## The before picture

Run the scanner view:

```
watcher report --raw
```

You get 56 findings. Highlights of the wall of text:

- 12 separate "Lambda function has s3:\*, dynamodb:\*, sqs:\* on all resources"
- 18 separate "pipeline authenticates with long-lived credential"
- 18 separate "principal holds AdministratorAccess"
- 4 separate "service reachable from internet on port 22/5432"
- assorted public-bucket alerts

This is the industry-standard experience. Every line is true. A team
triaging this will fix maybe the top three, argue about severity for the
rest, and the same 56 will be back next quarter — because nobody asked
*why there are twelve identical IAM findings*.

## The after picture

```
watcher report
```

6 root causes. The interesting one:

> **Shared over-permissioned IAM policy pattern (lambda-exec-standard)** —
> explains 12 findings, fix effort: days, touches 12 services

Expand its reasoning trace in the dashboard and you can read the actual
graph walk: the policy grants service wildcards on `Resource:*` → it's
attached to `lambda-exec-standard-role` → twelve Lambda functions assume
that role. The policy's own metadata records where it came from: a
Terraform module that's been copy-pasted since 2023.

That's the aha: **the fix is not twelve tickets. It's one pull request
against one Terraform module**, rolled out service-by-service with Access
Analyzer data so nothing under-scopes. The blast radius panel says exactly
that, including what breaks if you get it wrong.

## The part scanners structurally can't do

Two of Brightpath's root causes compound each other:

1. `AWS_DEPLOY_KEY` — one long-lived key, stored as a GitHub org secret,
   used by 18 repos' deploy jobs, mapping to `AdministratorAccess`.
2. `platform-core` — the service 13 others call directly.

A scanner sees these as unrelated line items (if it sees the second one at
all). On the graph they're one story: anyone with write access to any of
18 repos holds admin over the account that runs the service everything
depends on. That sentence — which is the *actual risk posture* of this
company — only exists if you model the relationships.

## Try it against something real

```
watcher ingest --github your-org --token ghp_...   # read-only
watcher ingest --aws --merge                       # read-only, scoped policy in docs/
watcher report
```

The connectors ingest less than the demo models (see honest-limitations in
the README), but the clustering works the same on real data: shared
policies, shared secrets, shared security groups are exactly the things
real estates copy-paste.
